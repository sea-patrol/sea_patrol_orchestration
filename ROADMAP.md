# ROADMAP.md — Sea Patrol

Дата: 2026-03-05

## 0) Видение (после MVP)
Цель: MMORPG про парусники с разными типами кораблей, **non-target** боёвкой, экономикой, ресурсами и крафтом, лором и квестами, разными режимами (арена/CTF и др.), с возможностью **горизонтального масштабирования бэкенда** и дальнейшего **шардирования мира**.

## 1) MVP (ближайшая минимальная цель)
MVP = **комнаты‑песочницы** (инстансы открытого моря), куда игрок попадает сразу.

### 1.1 Что должно быть в MVP (scope)
Игровой цикл «за 10 минут»:
1) зайти в игру → 2) плавать в комнате → 3) встретить NPC/игрока → 4) подраться (ядра + столкновения) → 5) подобрать/выловить ресурсы → 6) выполнить простой квест → 7) поторговать/пополнить припасы.

Фичи MVP:
- **Server-authoritative**: движение/боёвка/столкновения/лут — на сервере.
- Комнаты‑песочницы (создаются заранее) + задел под «режимные комнаты» (арена) *без реализации на первом этапе*.
- Типы кораблей (механизм смены/классы), но **визуально один glb‑шаблон**.
- NPC минимум: пират, торговец; животные: кит, рыба (вылов).
- Лут в море: «ящики» с провизией/ядрами/деревом/парусиной.
- Персистентность: аккаунт, корабль/тип корабля, деньги, трюм/ресурсы, инвентарь, квесты, позиция/прогресс (минимальный набор данных для “продолжить игру”).
- Смерть/потопление: **respawn без штрафа**.

### 1.2 Что НЕ входит в MVP (явно)
- Полноценный крафт/цепочки ресурсов, сложная экономика и торговля между игроками.
- Продвинутый лор/сюжетные цепочки квестов.
- Режимные комнаты (арены 1x1/2x2/5x5) — только архитектурные зацепки.
- Настоящее шардирование мира (zone/sector sharding) — только подготовка через room-based scaling.
- Броня/криты/сложные модификаторы урона.

## 2) Текущее состояние (снапшот)
Backend:
- WebFlux + WebSocket `/ws/game`, одна комната `main` логически возможна.
- Tick комнаты сейчас по умолчанию **100 мс (10 Hz)**: `GameRoom.DEFAULT_UPDATE_PERIOD_MILLIS = 100L`.
- Комнаты создаются лениво, лимита игроков нет: `GameService.rooms.computeIfAbsent(...)`.
- Пользователи/состояния — in-memory.

Frontend:
- React + R3F, WS‑клиент и адаптер сообщений есть.
- Уже есть зачатки HUD: `ProfileBlock`, `GameStateInfo`, `ChatBlock`, но пока без единой модели игрового интерфейса.
- Управление кораблём сейчас занято клавишами `W/A/S/D` и `ArrowUp/ArrowDown/ArrowLeft/ArrowRight`.

Документы:
- Оркестрация: `PROJECTS_ORCESTRATION_INFO.md`, `API.md`.
- Каноника подпроектов: `sea_patrol_backend/ai-docs/*`, `sea_patrol_frontend/ai-docs/*`.

## 3) Архитектурные принципы MVP
1) **Server-authoritative simulation**
   - Клиент отправляет только ввод/команды.
   - Сервер решает коллизии/попадания/лут/квестовые события.

2) **Room-based scaling (первый этап масштабирования)**
   - Комната = единица симуляции.
   - Горизонтальное масштабирование сначала как “распределение комнат по инстансам”.

3) **Интерфейсы вместо ранних интеграций**
   - Хранилище/кэш сначала in-memory/H2, но сразу через интерфейсы, чтобы позже включить Redis/Postgres без переписывания логики.

4) **Детерминированность и воспроизводимость** (насколько возможно)
   - Фиксируем tick‑rate/step, логируем события, держим понятные источники истины.

5) **UI shell отдельно от 3D-сцены**
   - Игровой HUD, окна и меню должны жить отдельным UI-слоем поверх canvas.
   - 3D-сцена не должна хранить бизнес-логику окон, инвентаря, карты и журнала.
   - Горячие клавиши UI должны проверяться на конфликты с управлением кораблём и чатом.

## 4) Дорожная карта (milestones)

Ниже — последовательность этапов. Порядок — рекомендованный для сохранения темпа и снижения рисков.

### M0 — Выравнивание контрактов + базовая инженерка
**Цель:** убрать рассинхрон фронт/бэк и создать “скелет” для дальнейших фич.

Backend:
- Уточнить и стабилизировать REST‑контракты auth (ответы/ошибки), чтобы фронт корректно показывал сообщения.
- Версионирование/совместимость WS‑протокола (минимум: договорённость формата и message types).
- Вынести конфиги комнаты (tick, лимиты) в конфиг/параметры (без подбора значений).

Frontend:
- Единый разбор ошибок auth (поддержка backend‑формата `errors[]`).
- Консолидация ожиданий по login/signup (не падать при отсутствии `userId`).
- Подготовить базовый `GameUiShell` поверх сцены как отдельный слой для HUD/окон/хоткеев.

Infra/Quality:
- Минимальная наблюдаемость: структурные логи + метрики tick/лаг/кол-во объектов (хотя бы на уровне логов).

**Критерии готовности:** фронт и бэк согласованы по auth/ошибкам, WS не ломается при небольших изменениях.

### M1 — Комнаты‑песочницы (управление и ограничения)
**Цель:** комнаты созданы/доступны игрокам, вход через лобби, есть лимиты и понятная модель жизненного цикла.

Backend:
- RoomRegistry: комнаты, управляемые сервером.
  - дефолты MVP: `maxRooms=5`, `maxPlayersPerRoom=100`.
  - Первая комната создаётся автоматически при старте (например `sandbox-1`).
  - Дополнительные комнаты могут создаваться по запросу игрока, но не выше `maxRooms`.
- Каталог комнат (MVP): список комнат + текущая заполненность (`current/maxPlayersPerRoom`) + информация о выбранной карте (`mapId`, `mapName`).
- Join/create policy:
  - если есть незаполненные комнаты — игрок выбирает и заходит;
  - если все заполнены — можно создать новую (если не достигнут `maxRooms`).
- Явная модель join/leave/cleanup + корректная обработка реконнекта.
- Rooms state в лобби (MVP): frontend при открытии страницы лобби делает `GET /api/v1/rooms` для первичного snapshot, затем сервер шлёт изменения списка комнат и заполненности через WS (`ROOMS_UPDATED`, опционально `ROOMS_SNAPSHOT` при реконнекте), без polling.
- REST (MVP): `GET /api/v1/rooms` для списка, `POST /api/v1/rooms` для создания комнаты, `POST /api/v1/rooms/{roomId}/join` для входа в комнату (enforce `maxRooms`, `maxPlayersPerRoom`, room admission rules).
- Room/world selection:
  - каждая комната создаётся из `map template` / `world definition`;
  - в MVP реально доступен только один шаблон карты, но механизм выбора закладывается сразу;
  - `POST /api/v1/rooms` должен уметь принимать `mapId` (опционально, с дефолтом на основную карту MVP);
  - backend валидирует `mapId` по реестру доступных карт и сохраняет выбор карты в состоянии комнаты.
- Валидация и rate-limit входящих команд (минимальный анти‑спам).
- Ветер/паруса (простая модель для MVP):
  - server-side `sailLevel` (дискретно): `0..3` (0=паруса убраны, 3=все паруса);
  - стартовое значение в MVP: `sailLevel=3`;
  - управление: `PLAYER_INPUT.up`/`PLAYER_INPUT.down` = +1/-1 уровня по rising-edge;
  - тяга вдоль курса: `F = sailLevelFactor * wind.speed * sailPower * eff(theta)`;
  - синхронизация: `sailLevel` приходит клиенту в `INIT_GAME_STATE` и/или `UPDATE_GAME_STATE`;
  - no-go zone: при `theta < noGoRad` эффективность снижается, но не ниже `minEff`.

Frontend:
- Лобби комнат: при открытии страницы выполняется `GET /api/v1/rooms`, затем список комнат (пример: `sandbox-1 (10/100)`), кнопки `join` и `create` (если доступно); у комнаты видны `mapName`/тип карты; дальнейшие изменения обновляются по WS событиям.
- Лобби: WebSocket подключается уже на этом экране для чата лобби (`to="group:lobby"`) и live-обновлений комнат; чат комнаты отдельный (`to="group:room:<roomId>"`). Вход в игровую комнату — через REST `POST /api/v1/rooms/{roomId}/join`, после чего сервер переключает WS-сессию на room streams.
- UX входа в комнату (информирование о реконнекте/потере связи).
- Отображение текущей комнаты/онлайна (минимально).

Дополнительные пожелания на доработку Wave 1 перед переходом к следующему большому этапу:
- Изоляция чатов должна быть видна не только на уровне transport contract, но и как реальное end-to-end поведение runtime/UI.
  - Lobby chat остаётся отдельным каналом `group:lobby`.
  - После входа в комнату игрок видит и отправляет сообщения только в чат своей комнаты `group:room:<roomId>`.
  - Сообщения из других комнат не должны попадать в клиент текущего игрока.
  - UI должен явно показывать, в каком chat scope сейчас находится пользователь: `Lobby` или конкретная `Room`.
- Лобби должно стать отдельной лёгкой HTML-страницей, а не overlay поверх уже загруженной 3D-сцены.
  - На странице лобби не должен монтироваться `Canvas` / R3F / gameplay scene.
  - Лобби должно открываться быстро и оставаться визуально простым: список комнат, create/join actions, статус соединения, lobby chat.
  - 3D-сцена и gameplay shell должны загружаться только после успешного `join` и перехода в room/game flow.
  - Навигация между `Home -> Lobby -> Room` должна быть явной и понятной.
- Первый слой in-game HUD:
  - слева сверху `PlayerStatusCard`;
  - слева снизу `ChatPanel`;
  - справа сверху `GameMenuButton`;
  - справа снизу `CompassMiniMapCluster`;
  - рядом с миникартой быстрые кнопки `Inventory` и `Journal`.

**Критерии готовности:** игрок всегда попадает в комнату, сервер не раздувает ресурсы при дисконнектах.

### M2 — Базовая боёвка (ядра + столкновения) и урон
**Цель:** “первый бой” работает и ощущается честным (server-authoritative).

Backend:
- Столкновения кораблей: урон от импакта (простая модель: скорость/масса → damage).
- Стрельба ядрами: минимальный протокол команд (fire/aim/reload).
- Модель снаряда с учётом ограничения: **физика 2D (плоскость XZ)**.
  Зафиксировано для MVP (вариант A: **server ray/segment cast** в Box2D):
  - Клиент → сервер: `FIRE_CANNON` (команда намерения)
    - payload: `{ side: "PORT"|"STARBOARD", aimOffsetRad: number }`
    - `aimOffsetRad` — отклонение от строго борта (перпендикуляра к курсу);
      `aimOffsetRad > 0` = «чуть к носу (вперёд по курсу)», сервер клампит (например ±0.35 rad).
  - Сервер (authoritative):
    - вычисляет `origin/dir` от позиции и ориентации корабля + выбранный борт;
    - добавляет разброс по углу (канониры) и джиттер скорости ("порох") — только на сервере;
    - симулирует снаряд в плоскости XZ как логическую сущность (без 3D физики);
    - на каждом update делает Box2D ray/segment cast `prevPos → nextPos` и берёт **первое** пересечение (без "пробития").
  - Сервер → клиенты: события (без стриминга траектории): `PROJECTILES_SPAWN` (залп), `PROJECTILE_HIT`, `PROJECTILE_DESPAWN`.
  - Координата Y: сервер **не считает**; клиент рисует косметическую дугу и обрывает по `PROJECTILE_HIT`.
  - Стартовые параметры (черновик): `cannonballsPerVolley=3`, `ttlMs≈2500`, `baseSpeed/spread/jitter` зависят от типа корабля.
- HP корабля, повреждения, “потопление” → respawn без штрафа.

Frontend:
- UI боёвки: перезарядка, индикатор HP, события попаданий.
- Визуализация выстрела/попаданий (даже если 3D “дуга” фейковая).
- Расширение `PlayerStatusCard`: HP корабля, статус парусов, боезапас, краткий combat state.

**Критерии готовности:** два игрока могут стабильно подраться ядрами/тараном, сервер подтверждает попадания.

### M3 — Взаимодействия мира: лут‑ящики, трюм, рыбалка, NPC (минимум)
**Цель:** появляется PvE/эксплор и ресурсы, которые дают смысл экономике/квестам.

Backend:
- Сущности мира: floating crates (spawn/respawn), рыба/кит как сущности.
- Подбор/вылавливание: server-authoritative, проверка дистанции/скорости/коллизий.
- Трюм корабля (cargo): вместимость, типы предметов (провизия/ядра/дерево/парусина/рыба).
- NPC:
  - Пират: простейший AI (патруль/агро/стрельба).
  - Торговец: фиксированный ассортимент и цены.

Frontend:
- UI трюма (минимум: список предметов + кол-во).
- Взаимодействие с ящиком/рыбалкой (prompts).
- Визуализация NPC/ящиков (пока условные модели/иконки).
- Первый usable inventory drawer/panel поверх HUD.

**Критерии готовности:** можно добыть ресурсы, положить в трюм, потратить их (стрельба/провизия/покупка).

### M4 — Персистентность (H2) + интерфейсы под будущие интеграции
**Цель:** “всё перечисленное сохраняется” и это не ломает reactive‑стек.

Backend:
- Выбрать реактивный доступ к данным:
  - Spring Data R2DBC + **H2** (file mode) для старта.
- Схема данных (v0, MVP; H2 → Postgres миграция без переписывания логики):
  - `users`: `id`, `username` (unique), `password_hash`, `email`, `created_at`
  - `ships`: `id`, `user_id`, `ship_type` (SLOOP/BRIG/GALLEON), `name`, базовые статы корабля, `created_at`
  - `user_state`: `user_id`, `active_ship_id`, `currency`, `last_room_id`, `last_x`, `last_z`, `updated_at`
  - `ship_cargo`: `ship_id`, `item_type`, `qty` (PK: `ship_id + item_type`)
  - `quests`: `user_id`, `quest_id`, `status`, `progress_json` (TEXT), `updated_at`
  - `map_templates`: `id`, `code` (unique), `name`, `region`, `revision`, `status`, `bounds_json`, `spawn_rules_json`, `meta_json`, `created_at`
  - `room_instances` (если решим хранить в БД не только активное runtime-состояние): `id`, `map_template_id`, `status`, `created_by`, `created_at`, `closed_at`
  - Правило: JSON используем только как TEXT для гибких полей (например `progress_json`/cosmetics), а core‑сущности держим нормализованными таблицами.
  - `item_type` (старт): `PROVISIONS`, `CANNONBALLS`, `WOOD`, `CLOTH`, `FISH`.
- Миграции схемы: только `Liquibase`, включая changelog для H2 и последующего Postgres.
- Абстракции:
  - `PlayerRepository`/`InventoryRepository`/`QuestRepository`
  - `RoomStateStore` (in-memory сейчас)
  - `MatchmakingStore` (in-memory сейчас)
  - Контракты, совместимые с будущим Redis/Postgres.

Frontend:
- Восстановление сессии/профиля после перезахода.
- Экран/виджет профиля (корабль, деньги, трюм, активный квест).
- Состояние открытых окон HUD не должно ломать восстановление игровой сессии.

**Критерии готовности:** после перезапуска сервера игрок сохраняет корабль/деньги/трюм/квесты.

### M5 — Квесты и простая экономика (замкнутый MVP‑луп)
**Цель:** игроку есть чем заниматься кроме “пострелять”.

Backend:
- Простые типы квестов:
  - принести ресурс (deliver)
  - поймать рыбу (catch)
  - потопить пирата (kill)
- Награды: деньги/ресурсы.
- Торговец: покупка/продажа (простая формула цен), пополнение припасов.

Frontend:
- UI квестов: активный квест + прогресс.
- UI торговли: купить/продать, отображение баланса.
- Судовой журнал как отдельное окно:
  - активный квест;
  - история последних событий/логов;
  - записи о найденных ресурсах/NPC/точках интереса (минимально, без энциклопедии).
- Полноэкранная карта мира/комнаты с опорой на миникарту и компас.

**Критерии готовности:** есть повторяемый цикл “добыча → квест → награда → улучшение снабжения”.

### M6 — Подготовка к горизонтальному масштабированию (без шардирования мира)
**Цель:** добавить точки расширения, чтобы позже распределять комнаты между инстансами.

Backend:
- Отвязать “комната” от процесса:
  - модель назначения комнаты на инстанс (instanceId)
  - интерфейс координации (пока in-memory)
- Вебсокеты: предусмотреть sticky routing / выдачу игроку конкретного WS‑адреса комнаты.
- Интерфейс для распределённого кэша (будущий Redis) и блокировок.

Infra:
- Документировать деплой “несколько инстансов” (даже без реальной балансировки).
- План наблюдаемости: метрики tick, queue, dropped updates, количество сущностей.

**Критерии готовности:** архитектура допускает включение Redis и разделение комнат по инстансам без переписывания core‑логики.

## 5) Post‑MVP (вектор развития)
- Режимные комнаты (арены 1x1/2x2/5x5), матчмейкинг, готовность игроков.
- Расширение кораблей: реальные разные модели/хитбоксы/классы, прогрессия.
- Крафт и ресурсные цепочки.
- Лор и цепочки квестов.
- Полноценная экономика (аукцион/торговля между игроками).
- Переход H2 → Postgres; подключение Redis.

  - зоны/сектора, миграция игрока между секторами
  - сервисы “world/zone”, “combat”, “economy”, “quest” (по мере необходимости)

## 6) Риски и неизвестные (обязательные spikes)
1) **Лимиты движка/симуляции (Box2D + количество объектов)**
   - Spike: нагрузочный тест комнаты (N игроков + M NPC + K снарядов) → CPU/RAM/latency.

2) **2D физика vs 3D визуализация (баллистика/земля/берег)**
   - Spike: реализовать вариант A (server ray/segment cast) и подобрать параметры залпа/разброса/скорости/TTL.
   - Spike: модель суши (простые 2D коллайдеры островов) и взаимодействие.

3) **Сетевой протокол и частоты обновления**
   - Сейчас 10 Hz; возможно потребуется поднять частоту или перейти на фиксированный step + интерполяцию.

4) **Персистентность “всего”**
   - Данные должны быть минимально достаточными; часть “позиции” возможно лучше хранить как порт/спавн, а не точные координаты.

5) **Нагрузка интерфейса и конфликты ввода**
   - Нужно отделить hotkeys управления кораблём от hotkeys UI/окон и от режима ввода чата.
   - Любое открытие окна не должно ломать управление кораблём неожиданным образом.

## 7) Что обсудить дальше (следующие решения)
1) Параметры ядер/залпа для MVP: `cannonballsPerVolley`, `ttlMs`, `baseSpeed`, `spreadRad`, `speedJitter` (и их зависимость от типа корабля).
2) “Суша” в MVP: нужна ли уже сейчас (острова/берег/порт), и какая механика взаимодействия.
3) Набор типов кораблей (3 класса) и их ключевые параметры (скорость/манёвренность/HP/трюм/`cannonsPerSide`/перезарядка/`cannonballsPerVolley`).
4) Правила трюма: вместимость, расход провизии, расход ядер, ремонт (дерево/парусина).
5) Минимальный UI: трюм/квесты/торговля/боёвка.
6) UI-состояния: какие окна блокируют управление кораблём, а какие остаются поверх игрового процесса.

## 8) Зависимости между milestone'ами
Чтобы не распараллелить работу ошибочно, ниже фиксируется рекомендуемая зависимость этапов:

- `M0` обязателен перед активной разработкой `M1`, потому что комнаты и лобби сразу упираются в согласованный auth/WS-контракт.
- `M1` обязателен перед `M2`, `M3`, `M5`, потому что боёвка, NPC, лут и квесты должны жить в явной модели комнаты, а не в текущем неявном `main`.
- `M2` частично независим от `M4`, но до релиза MVP должен быть совместим с будущей персистентностью (`ship hp`, ресурсы, respawn state).
- `M3` и `M5` разумно вести после стабилизации `M2`, иначе интерфейсы cargo/loot/trading начнут меняться на ходу.
- `M4` можно начинать после фиксации доменной модели `M2` + `M3`, иначе схема данных будет быстро переписываться.
- `M6` не нужен для первого игрового цикла, но интерфейсы координации комнат надо подготавливать уже в `M1`, чтобы не вшивать room lifecycle в один процесс слишком глубоко.
- HUD/UI-shell нужно закладывать уже в `M0-M1`, чтобы потом не переделывать слой поверх canvas при добавлении карты, журнала и инвентаря.

Итоговая цепочка для MVP:
`M0 -> M1 -> M2 -> M3 -> M4 -> M5`, при этом заделы для `M6` закладываются в `M1` и `M4`.

## 9) Ближайший delivery plan (рекомендуемый порядок)
Ниже — не просто milestone'ы, а конкретная очередь работ, которую уже можно резать на задачи.

### Wave A — выравнивание контрактов и лобби-комнат
`Цель:` закрыть все блокеры перед переходом к нескольким комнатам.

Orchestration:
- Зафиксировать единый auth-контракт для `signup/login/error response`.

- Отдельно описать expected join flow: `GET /api/v1/rooms` -> lobby WS/chat connect -> REST `POST /api/v1/rooms/{roomId}/join` -> room state init.

Эта секция нужна, чтобы из roadmap можно было быстро создать backlog без повторного анализа.

## 11) Definition of Ready / Definition of Done для roadmap-задач
Чтобы не заводить расплывчатые задачи, фиксируем минимальные критерии.

### Definition of Ready
Задача готова к реализации, если:
- ясно, в каком репозитории находится source of truth;
- описан внешний контракт задачи: REST, WS message, UI state или schema change;
- понятны зависимости на предыдущие milestone'ы;
- есть критерий проверки: тест, ручной сценарий или документируемый expected flow.

### Definition of Done
Задача считается завершенной, если:
- код и документация не противоречат друг другу;
- затронутые REST/WS контракты обновлены в orchestration и, при необходимости, в подпроектах;
- есть минимум одна проверка на уровень изменения:
  - backend: unit/integration test или явно описанный ручной сценарий;
  - frontend: test/build/manual scenario для UI-потока;
  - orchestration: обновлены связанные sections в `API.md`, `PROJECTS_ORCESTRATION_INFO.md`, `ROADMAP.md`;
- результат можно встроить в следующий milestone без дополнительного `разбора полетов`.

## 13) Frontend HUD / UI layer (предлагаемая каноника)
Эта секция фиксирует целевую раскладку игрового интерфейса на фронте.

### 13.1 Базовый layout HUD
- **Слева сверху**: `PlayerStatusCard`.
  - Аватар игрока.
  - Имя/ник.
  - Уровень/ранг.
  - HP корабля.
  - Дополнительные быстрые характеристики MVP: скорость, состояние парусов, боезапас, вместимость трюма или золото.
- **Слева снизу**: `ChatPanel`.
  - Отдельные режимы/каналы: минимум `lobby`, `room`, `system`.
  - Важное правило: при активном фокусе в поле ввода игровые hotkeys не должны срабатывать.
- **Справа сверху**: `GameMenuButton`.
  - Открывает игровое меню/пауза-overlay.
  - Рядом можно показывать маленькие иконки/подсказки хоткеев.
- **Справа снизу**: `CompassMiniMapCluster`.
  - Компас как постоянный элемент.
  - Миникарта как компактный круг/квадрат рядом с компасом.
  - Клик по миникарте открывает полную карту, если она уже реализована.
- **Слева от блока компаса/миникарты**: быстрые иконки `Inventory` и `Journal`.
  - Это отдельные кнопки HUD, а не часть меню.
  - При наведении/тапе должны показывать хоткей (`I`, `J`).

### 13.2 Окна и панели
- `InventoryWindow`:
  - содержимое трюма/инвентаря;
  - вес/вместимость;
  - ресурсы и расходники.
- `JournalWindow`:
  - активный квест;
  - история последних событий;
  - заметки по миру/точкам интереса в упрощённом виде.
- `MapWindow`:
  - карта текущей комнаты или мира;
  - позиция игрока;
  - базовые маркеры комнатных POI/NPC/порта, когда они появятся.
- `GameMenuOverlay`:
  - настройки;
  - выход в лобби/на главную;
  - справка по управлению.

### 13.3 Правила поведения окон
- `ChatPanel`, `PlayerStatusCard`, `GameMenuButton`, `CompassMiniMapCluster` всегда видимы в боевом режиме.
- `InventoryWindow`, `JournalWindow`, `MapWindow`, `GameMenuOverlay` открываются поверх HUD.
- Одновременно по умолчанию открыто только одно крупное окно из набора: `Inventory`, `Journal`, `Map`, `Menu`.
- Открытие крупного окна должно либо:
  - ставить корабль в безопасный `UI focus mode`,
  - либо явно оставлять корабль управляемым; для MVP рекомендуется **блокировать игровое управление**, пока открыто крупное окно.
- Чат не считается крупным окном; он может оставаться доступным независимо.

### 13.4 Предлагаемые hotkeys (MVP)
- `Esc` — открыть/закрыть `GameMenuOverlay`.
- `M` — открыть/закрыть `MapWindow`.
- `J` — открыть/закрыть `JournalWindow`.
- `I` — открыть/закрыть `InventoryWindow`.
- `Enter` — перевести фокус в чат; повторный `Enter` отправляет сообщение.
- `Esc` при фокусе в чате — снять фокус с чата, не отправляя сообщение.

Причины выбора:
- `W/A/S/D` и стрелки уже заняты под корабль.
- `M`, `J`, `I`, `Esc` — привычные игровые клавиши и не конфликтуют с текущим движением.
- `Tab` лучше пока не занимать, чтобы оставить его под список игроков, scoreboard или cycle-target в будущем.

### 13.5 MVP vs позже
В MVP frontend-слой интерфейса должен включать:
- `PlayerStatusCard`;
- `ChatPanel`;
- `GameMenuButton` и `GameMenuOverlay`;
- `CompassMiniMapCluster`;
- кнопки и hotkeys для `Inventory`, `Journal`, `Map`.

При этом содержимое окон может вводиться поэтапно:
- сначала каркас окон и переключение по hotkeys;
- затем инвентарь;
- затем журнал;
- затем полнофункциональная карта.





## 14) Игровые миры / карты (предлагаемая каноника)

### 14.1 Базовая модель
- Нужно различать `map template` и `room instance`.
- `Map template` — это шаблон мира: Карибы, Японские острова и т.п.
- `Room instance` — конкретная созданная комната, которая использует один `map template`.
- Для MVP допускается только один шаблон карты, но API и backend-модель должны быть рассчитаны на несколько шаблонов.

### 14.2 Что должно выбираться при создании комнаты
- При `POST /api/v1/rooms` клиент должен иметь возможность передать `mapId`.
- Если `mapId` не передан, backend использует дефолтный шаблон карты MVP, например `caribbean-01`.
- В ответе на `GET /api/v1/rooms` и в room catalog должны присутствовать минимум:
  - `mapId`;
  - `mapName`;
  - опционально `mapPreview`/`region`, если это понадобится для UI лобби.

### 14.3 Как описывать карту
Для gameplay-логики карту лучше описывать как набор данных, а не как одну 3D-модель.

Рекомендуемое разбиение:
- `manifest` — метаданные карты: `id`, `name`, `region`, `revision`, границы мира, дефолтные правила ветра/погоды.
- `spawn points` — точки/зоны спавна игроков, NPC, ресурсов.
- `navigation/blocking geometry` — упрощённая серверная геометрия: острова, берега, запретные зоны, коллайдеры.
- `points of interest` — порты, торговцы, квестовые точки, landmarks.
- `visual asset refs` — ссылки на клиентские ассеты: terrain/ocean/skybox/props/minimap overlay.
- `rules overrides` — опциональные правила конкретной карты: лимиты NPC, ветер, типы ресурсов, специальные модификаторы.

### 14.4 Где хранить карту
Для ближайших этапов лучший компромисс такой:
- Источник истины для самих шаблонов карт хранить в файлах репозитория backend, например `src/main/resources/worlds/<mapId>/...`.
- В БД хранить реестр карт (`map_templates`) и ссылки/метаданные, которые нужны приложению и админским сценариям.
- В runtime backend загружает `MapTemplateRegistry` на старте из ресурсов и/или сверяет его с БД.

Почему так:
- геометрию, spawn-зоны и POI удобнее версионировать файлами вместе с кодом;
- БД не лучший первый дом для больших JSON/геометрий и игровых ассетов;
- при появлении редактора карт позже можно будет добавить импорт в БД/Object Storage, не ломая текущую модель.

### 14.5 Рекомендуемая структура файлов карты
Пример для одной карты:
- `src/main/resources/worlds/caribbean-01/manifest.json`
- `src/main/resources/worlds/caribbean-01/colliders.json`
- `src/main/resources/worlds/caribbean-01/spawn-points.json`
- `src/main/resources/worlds/caribbean-01/poi.json`
- `src/main/resources/worlds/caribbean-01/minimap.json` или `minimap.png` + calibration metadata

Если данных станет больше, можно перейти на формат:
- `world.yaml/json` как корневой manifest;
- дочерние файлы по доменам: `navigation`, `spawns`, `poi`, `economy`, `quests`.

### 14.6 Что хранить в БД, а что не хранить
В БД стоит хранить:
- реестр доступных карт (`map_templates`);
- версию шаблона (`revision`);
- включена ли карта в matchmaking/create room;
- метаданные для UI лобби;
- связь комнаты с `map_template_id`, если состояние комнаты нужно сохранять вне runtime.

В БД пока не стоит хранить:
- тяжёлые 3D-ассеты;
- детальную геометрию визуального мира;
- всё, что удобнее версионировать как файлы конфигурации рядом с кодом.

### 14.7 Практическая рекомендация для MVP
- Сделать один шаблон карты: `caribbean-01`.
- Сразу ввести `mapId` в модель комнаты и в REST create/list contracts.
- Реализовать `MapTemplateRegistry`, даже если он возвращает только одну карту.
- Описывать карту файлами в ресурсах backend, а `Liquibase` использовать для схемы БД и сидирования реестра `map_templates`.
- Не пытаться в MVP делать полноценный редактор карт или хранение всей карты в таблицах.

### 14.8 Runtime-механизм загрузки шаблонов карт
На стороне backend нужен явный слой `MapTemplateRegistry` / `WorldTemplateService`.

Его обязанности:
- при старте приложения сканировать `src/main/resources/worlds/*`;
- читать `manifest` и связанные файлы шаблона карты;
- валидировать целостность шаблона до публикации в runtime;
- собирать in-memory реестр доступных карт по `mapId`;
- предоставлять API вида:
  - `getDefaultMap()`;
  - `getById(mapId)`;
  - `listAvailableMaps()`;
  - `validateTemplate(mapId)`.

Ожидаемое поведение:
- если шаблон карты битый, он не должен публиковаться в список доступных для создания комнат;
- если сломана дефолтная карта MVP, backend должен падать на старте, а не работать в частично валидном состоянии;
- список карт в lobby/create room должен формироваться из runtime-реестра, а не вручную из фронта.

### 14.9 Предлагаемый состав `manifest.json`
Минимум для MVP:
```json
{
  "id": "caribbean-01",
  "name": "Caribbean Islands",
  "region": "caribbean",
  "revision": 1,
  "enabled": true,
  "bounds": { "minX": -5000, "maxX": 5000, "minZ": -5000, "maxZ": 5000 },
  "spawnRules": {
    "playerSpawnMode": "POINTS",
    "npcSpawnEnabled": true,
    "resourceSpawnEnabled": true
  },
  "files": {
    "colliders": "colliders.json",
    "spawnPoints": "spawn-points.json",
    "poi": "poi.json",
    "minimap": "minimap.json"
  },
  "presentation": {
    "previewImage": "preview.png",
    "theme": "tropical"
  }
}
```

Пояснения:
- `id` — стабильный идентификатор шаблона карты, который используется в API и в БД;
- `revision` — версия шаблона, чтобы понимать, менялась ли карта между релизами;
- `bounds` — границы мира для сервера, миникарты и базовой валидации координат;
- `files` — явные ссылки на все дочерние data-файлы;
- `presentation` — не gameplay-данные, а информация для lobby/UI.

### 14.10 Разделение ответственности между файлами и БД
Файлы в `resources/worlds/<mapId>/...`:
- полное описание шаблона карты;
- геометрия коллайдеров;
- spawn-зоны;
- POI;
- данные для миникарты;
- любые map-specific gameplay overrides.

БД (`map_templates`):
- каталог карт, доступных приложению;
- краткие метаданные для API и UI;
- статус `ACTIVE/DISABLED/DRAFT`;
- `revision`, с которой карта зарегистрирована;
- техническая информация для сверки с runtime, например `content_hash` или `manifest_checksum`.

Ключевая мысль:
- файлы отвечают за содержимое карты;
- БД отвечает за реестр, включение/выключение и операционную видимость карты в приложении.

### 14.11 Как синхронизировать файл шаблона и запись в БД
Для MVP не нужен двусторонний sync-движок. Достаточно простого правила:
- source of truth для содержимого карты — файлы в backend-репозитории;
- `Liquibase` сидирует/обновляет базовый реестр `map_templates`;
- при старте backend читает файл и БД, затем проверяет, что:
  - `mapId/code` совпадают;
  - `revision` совпадает;
  - карта помечена как `enabled/active`, если должна быть доступна в lobby/create room.

Если есть расхождение:
- для MVP логировать ошибку и исключать карту из списка доступных;
- для дефолтной карты приложения считать это startup-blocker.

Это проще, чем пытаться хранить полную карту в БД и решать конфликт двух источников истины уже сейчас.

### 14.12 Flow создания комнаты с шаблоном карты
Рекомендуемый backend flow:
1. Клиент вызывает `GET /api/v1/rooms` и/или `GET /api/v1/maps` в будущем.
2. Клиент выбирает `mapId` при создании комнаты.
3. Клиент отправляет `POST /api/v1/rooms { mapId, ... }`.
4. Backend проверяет:
   - существует ли `mapId` в `MapTemplateRegistry`;
   - доступна ли карта для создания комнаты;
   - не превышены ли room limits.
5. Backend создаёт `room instance`, связывает его с `mapId` и инициализирует room state на основе шаблона:
   - world bounds;
   - spawn points;
   - initial wind/weather overrides;
   - resource/NPC spawn configuration.
6. Новая комната появляется в room catalog и отдаётся через REST/WS.

Это важно: комната хранит ссылку на шаблон карты, а не копию всех исходных map-файлов в памяти в сыром виде.

### 14.13 Что именно материализуется в room instance
Из шаблона карты в runtime комнаты должны материализоваться только нужные для симуляции данные:
- границы мира;
- подготовленные коллайдеры/навигационная геометрия;
- точки спавна игроков;
- runtime-конфигурация ресурсов и NPC;
- POI/landmarks, если они участвуют в gameplay или UI.

Не нужно копировать в каждую комнату:
- raw preview assets;
- исходные json-файлы как строки;
- всё, что относится только к frontend presentation и не нужно серверу на tick/update.

### 14.14 Версионирование шаблонов карт
Нужно различать:
- `mapId` — стабильный идентификатор шаблона (`caribbean-01`);
- `revision` — номер версии содержимого карты.

Рекомендация:
- minor правки контента той же карты повышают `revision`;
- принципиально новая карта получает новый `mapId`;
- активные комнаты живут на той `revision`, с которой были созданы, если не реализован механизм live-migration.

Для MVP:
- live-migration не нужна;
- изменение карты применяется только к новым комнатам после деплоя.

### 14.15 Валидация шаблона карты
Минимальные проверки на старте backend:
- все обязательные файлы шаблона существуют;
- `bounds` заданы корректно;
- есть хотя бы одна валидная player spawn point/zones;
- colliders не выходят за границы мира;
- POI имеют уникальные `id` внутри карты;
- `manifest.revision` и запись в `map_templates` согласованы.

Желательно добавить отдельный validation mode/test suite для карт, чтобы шаблон можно было проверить без запуска всей игры.

### 14.16 Нужно ли хранить карты целиком в БД
На текущем этапе — нет.

Почему:
- это усложнит миграции и review изменений;
- json/геометрия карт будут плохо жить как changelog-данные в `Liquibase`;
- для больших карт быстро появится соблазн складывать в БД то, что лучше лежит в ресурсах или object storage.

Когда это может измениться:
- если появится web-based editor карт;
- если нужен runtime upload/import карт без релиза backend;
- если появится отдельный world authoring pipeline.

Но это уже post-MVP сценарий.

### 14.17 Что можно добавить в roadmap дальше
После текущего уровня детализации логично будет отдельно зафиксировать:
- REST-контракт `POST /api/v1/rooms` с `mapId`;
- опциональный `GET /api/v1/maps` для списка доступных шаблонов карт;
- формат `manifest.json` и validation rules в backend `ai-docs`;
- минимальный пакет файлов для первой карты `caribbean-01`.

## 15) Spawn / Session / Room lifecycle (MVP policy)

### 15.1 Spawn и respawn policy
- Для MVP используем простой тестовый spawn rule: игрок появляется около центра карты.
- Канонически, несмотря на простоту правила, координаты спавна должен определять **backend**, а frontend только применять полученные координаты. Это сохраняет server-authoritative модель.
- Для MVP backend должен отправлять **отдельное сообщение спавна** с координатами и ориентацией корабля; клиент не вычисляет spawn самостоятельно.
- Базовая формула MVP:
  - базовая точка спавна: `(0, 0)` по плоскости `XZ`;
  - случайное смещение: `x in [-30, 30]`, `z in [-30, 30]`;
  - итоговая точка должна проходить валидацию по границам карты и blocking geometry.
- Для первой тестовой карты достаточно одной центральной spawn zone.
- Respawn после потопления:
  - происходит в той же комнате, если она ещё активна;
  - использует то же правило центрального спавна со случайным смещением;
  - штрафы в MVP не применяются;
  - HP/боезапас/провизия после respawn восстанавливаются по отдельному стартовому правилу MVP (нужно зафиксировать позже, но для MVP допустим полный reset ship combat state).

### 15.1.1 Spawn transport contract (planned MVP)
- Рекомендуемый отдельный transport event: `SPAWN_ASSIGNED`.
- Минимальный payload:
  - `roomId`
  - `reason: INITIAL | RESPAWN`
  - `x`
  - `z`
  - `angle`
- Порядок для initial spawn:
  1. room join подтверждён;
  2. backend вычисляет spawn point;
  3. backend отправляет `SPAWN_ASSIGNED`;
  4. backend отправляет `INIT_GAME_STATE` / последующий room snapshot.
- Порядок для respawn:
  1. сервер фиксирует смерть/потопление;
  2. рассчитывает новый spawn;
  3. отправляет `SPAWN_ASSIGNED { reason: RESPAWN }` или эквивалентный payload;
  4. клиент переставляет локальный корабль по authoritative координатам.

### 15.2 Геометрия объектов карты и мира
- Для MVP каждый объект игрового мира имеет простую 2D-форму в плоскости `XZ`:
  - `CIRCLE`
  - `RECTANGLE`
- Это относится к:
  - POI;
  - ресурсным объектам;
  - торговцам;
  - тестовым интерактивным зонам;
  - blocking/navigation geometry островов и берегов.
- Размеры и координаты объектов задаются данными карты.
- На первом этапе допустимо использовать тестовый шаблон карты с несколькими тестовыми объектами и условными размерами.
- Более сложные формы, полигоны и составные коллайдеры — post-MVP или отдельный spike.

### 15.3 Realtime session policy
- Один пользователь = одна игровая сессия.
- Несколько параллельных вкладок/клиентов для одного пользователя в MVP не допускаются.
- Duplicate login policy:
  - если у пользователя уже есть активная игровая сессия, новый логин/новое подключение должно отклоняться;
  - допустимое исключение: reconnect в рамках окна восстановления сессии.
- Disconnect grace period:
  - после потери соединения сессия удерживается `30 секунд`;
  - в это время игрок не удаляется мгновенно из runtime-состояния комнаты;
  - комната не должна считаться пустой, пока есть игроки в окне reconnect grace.

### 15.4 Reconnect policy (предложение для MVP)
- Если пользователь теряет WS-соединение, его сессия переводится в состояние `DISCONNECTED_GRACE` на `30 секунд`.
- Если тот же пользователь возвращается в течение этого окна:
  - сервер связывает новое соединение с существующей игровой сессией;
  - игрок восстанавливается в той же комнате;
  - позиция, HP, cargo и текущее room state сохраняются как были на момент дисконнекта;
  - сервер повторно шлёт актуальный snapshot комнаты (`ROOM_JOINED`/`INIT_GAME_STATE` или эквивалентный resume flow).
- Если окно `30 секунд` истекло:
  - старая игровая сессия считается завершённой;
  - игрок удаляется из runtime комнаты;
  - при следующем входе пользователь попадает в лобби как в новую игровую сессию.
- Для MVP лучше не делать отдельный сложный resume API; достаточно, чтобы reconnect использовал тот же auth principal и приводил к восстановлению существующей session binding на backend.

### 15.5 Room lifecycle policy
- Комната существует, пока в ней есть:
  - активные игроки;
  - или игроки в reconnect grace period.
- Пустая комната закрывается, когда:
  - в ней нет активных игроков;
  - и нет игроков в окне `DISCONNECTED_GRACE`.
- После закрытия комнаты:
  - runtime state освобождается;
  - комната удаляется из активного room registry;
  - она исчезает из lobby room catalog.
- Для MVP допустимо не сохранять полное runtime-состояние комнаты при закрытии.

### 15.6 Save policy игрока
- Пока runtime-состояние игрока живёт в памяти процесса.
- Персистентное сохранение в БД для MVP рекомендуется делать:
  - периодически, например каждые `30 секунд`;
  - при disconnect;
  - при успешном leave/room shutdown;
  - после значимых событий: trade, quest completion, изменение cargo/денег.
- Для MVP приоритетнее сохранить корректность и предсказуемость, чем пытаться писать в БД на каждом tick.

## 16) Interaction model и lifecycle сущностей (MVP policy)

### 16.1 Минимальная модель взаимодействий
Для MVP у любого интерактивного объекта должен быть единый каркас:
- `entityId`
- `entityType`
- `shape`
- `position`
- `interactionRadius` или геометрия взаимодействия
- `state`
- `allowedInteractions`

Минимальные типы взаимодействий MVP:
- `PICKUP`
- `FISH`
- `TRADE_OPEN`
- `QUEST_OPEN`
- `ATTACK`
- `COLLIDE`

Правило:
- клиент только инициирует действие;
- сервер проверяет дистанцию, состояние объекта, cooldown и применимость действия;
- результат взаимодействия всегда authoritative на сервере.

### 16.2 Минимальный lifecycle сущностей
Для большинства world entities достаточно такого базового lifecycle:
- `DEFINED_IN_TEMPLATE`
- `SPAWNED_IN_ROOM`
- `ACTIVE`
- `INTERACTED` / `DAMAGED` / `CONSUMED`
- `DESPAWNED`
- опционально `RESPAWN_SCHEDULED`

### 16.3 Lifecycle по основным типам сущностей
Игрок:
- `CONNECTED`
- `SPAWNED`
- `ACTIVE`
- `DISCONNECTED_GRACE`
- `REMOVED`

Projectile:
- `SPAWNED`
- `ACTIVE`
- `HIT` или `EXPIRED`
- `DESPAWNED`

Лут-ящик / ресурс:
- `SPAWNED`
- `ACTIVE`
- `PICKED`
- `RESPAWN_SCHEDULED`
- `RESPAWNED`

NPC:
- `SPAWNED`
- `ACTIVE`
- `AGGRO` или `INTERACTABLE`
- `DEAD` / `DESPAWNED`
- `RESPAWN_SCHEDULED`

Торговец / статичный POI:
- `SPAWNED`
- `ACTIVE`
- обычно не уничтожается и не респавнится в рамках MVP

### 16.4 Что важно не забыть для MVP
- Любая сущность должна иметь стабильный `entityId` в рамках жизни комнаты.
- Клиент должен уметь обрабатывать минимум события: `spawn`, `update`, `despawn`.
- Для MVP не нужно сразу проектировать универсальную ECS-архитектуру; достаточно единых правил lifecycle и transport contracts.

## 17) Static game data catalogs (MVP skeleton)

### 17.1 Каталог кораблей

Для MVP:

### 17.2 Каталог предметов
Минимальный набор item types:
- `GOLD`

- `CANNONBALLS`

Опционально, если нужен технический буфер для ремонта:

### 17.3 Каталог тестовых квестов
Минимальный каркас:

Static data:
- ship classes;
- item catalog;
- merchant catalog;
- quest definitions;
- map templates;

Runtime data:
- активная комната;

## 18) Минимальная логика торговли (MVP policy)

### 18.1 Где разрешена торговля
- Торговать можно только у торговца/в торговой точке карты.
- Торговая точка задаётся как POI или entity на карте.
- Для открытия торговли сервер проверяет:
  - дистанцию до торговца;
  - что игрок находится в допустимой зоне взаимодействия;
  - что игрок не находится в состоянии `DEAD` / `RESPAWN_PENDING`.

### 18.2 Упрощённые правила торговли для MVP
- Цены фиксированные, без динамического рынка.
- Остатки торговца бесконечные или условно бесконечные.
- Торговля происходит мгновенно без анимаций и без подтверждения второй стороны.
- Нет торговли между игроками.
- Нет аукциона.
- Нет торга, репутации и региональных modifiers.

### 18.3 Что можно покупать и продавать в MVP
Покупка у торговца:
- `PROVISIONS`
- `CANNONBALLS`
- `WOOD`
- `CLOTH`

Продажа торговцу:
- `FISH`
- `WOOD`
- `CLOTH`
- при необходимости часть лута/ящиков в будущем, но это не обязательно для первого MVP.

### 18.4 Серверная валидация торговли
Сервер обязан проверить:
- хватает ли у игрока `GOLD` на покупку;
- хватает ли места в трюме на покупку;
- есть ли у игрока нужный ресурс в достаточном количестве для продажи;
- находится ли игрок в допустимой точке торговли.

### 18.5 Результат торговли
После успешной операции сервер обновляет:
- баланс `GOLD`;
- cargo игрока;
- при необходимости quest progress;
- профиль/состояние игрока в persistence очереди.

## 19) UI mode model (MVP policy)

### 19.1 Набор UI modes
Для MVP достаточно формализовать следующие режимы:
- `LOADING`
- `LOBBY`
- `SAILING`
- `CHAT_FOCUS`
- `WINDOW_FOCUS`
- `MENU_OPEN`
- `RECONNECTING`
- `RESPAWN`

### 19.2 Что означает каждый режим
`LOADING`
- идёт загрузка страницы, auth/session restore или room init.
- Игровой input заблокирован.

`LOBBY`
- игрок находится на странице выбора комнат.
- Доступны lobby chat, create/join room, список комнат.

`SAILING`
- основной игровой режим.
- Активно управление кораблём.
- HUD видим, крупные окна закрыты.

`CHAT_FOCUS`
- фокус в поле ввода чата.
- Игровые hotkeys не должны срабатывать.

`WINDOW_FOCUS`
- открыто крупное окно (`Inventory`, `Journal`, `Map`).
- Для MVP управление кораблём блокируется.

`MENU_OPEN`
- открыто игровое меню.
- Управление кораблём блокируется.

`RECONNECTING`
- соединение потеряно, клиент пытается восстановить session binding.
- UI показывает статус восстановления; активный input ограничен.

`RESPAWN`
- игрок потоплен и ждёт восстановления/повторной инициализации.
- Обычный sailing input заблокирован.

### 19.3 Базовые переходы между режимами
- `LOADING -> LOBBY`
- `LOBBY -> LOADING -> SAILING` при успешном join комнаты
- `SAILING -> CHAT_FOCUS` по `Enter`
- `CHAT_FOCUS -> SAILING` по `Esc` или после отправки сообщения
- `SAILING -> WINDOW_FOCUS` по `I/J/M`
- `WINDOW_FOCUS -> SAILING` по `I/J/M/Esc`
- `SAILING -> MENU_OPEN` по `Esc`
- `MENU_OPEN -> SAILING` по `Esc`
- `SAILING -> RECONNECTING` при потере соединения
- `RECONNECTING -> SAILING` при успешном reconnect
- `RECONNECTING -> LOBBY` или `LOADING` если reconnect grace истёк
- `SAILING -> RESPAWN` при потоплении
- `RESPAWN -> SAILING` после respawn init

### 19.4 Ограничения для реализации
- Одновременно активен только один крупный UI mode поверх `SAILING`.
- `CHAT_FOCUS` и `WINDOW_FOCUS` не должны пересекаться.
- `RECONNECTING` имеет приоритет над обычными окнами.
- `MENU_OPEN` должен закрывать `WINDOW_FOCUS` или не открываться поверх него без явного правила.

## 20) Explicit seed strategy (минимально необходимая)

### 20.1 Что нужно сидировать обязательно
Для пустой БД MVP должен подниматься с минимальным набором данных:
- default map template: `caribbean-01`;
- test map template: `test-sandbox-01`;
- ship classes: `SLOOP`, `BRIG`, `GALLEON`;
- item catalog;
- merchant starter catalog;
- quest definitions: минимум 3 тестовых квеста;
- при необходимости NPC archetypes: `PIRATE_BASIC`, `MERCHANT_BASIC`, `WHALE_BASIC`, `FISH_SCHOOL_BASIC`.

### 20.2 Как сидировать
- `Liquibase` changelog делится минимум на:
  - schema changelog;
  - reference data changelog;
  - optional dev/test seed changelog.
- Seed-данные должны быть идемпотентными.
- Production и dev seed можно разделять профилями или отдельными changelog include.

### 20.3 Что не нужно сидировать на MVP старте
- активные room instances;
- живые runtime сущности мира;
- пользовательские аккаунты, если это не отдельный dev profile;
- большие JSON-карты целиком через SQL inserts.

### 20.4 Практическое правило
Если данные нужны, чтобы пустой стенд после запуска сразу позволял:
- создать комнату;
- выбрать карту;
- заспавнить игрока;
- пострелять;
- подобрать ресурс;
- поторговать;
- взять тестовый квест,
то эти данные должны либо сидироваться, либо грузиться из backend resources как guaranteed defaults.
